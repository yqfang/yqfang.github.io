那些关于定时迭代的事儿

Jun 9, 2017 | js | Hits

前言

定时迭代数组 `[1, 2, 3, 4]` 按照顺序每秒 console 一个元素。这是道 javascript 的经典面试题。
我们来说说关于它的一些解法和背后的故事。

很耿直的解法

这段代码有几个问题：

  1. 每个变量，包括 i 都暴露在全局作用域下，污染了 window 变量
  2. 定时处理函数中存在函数外的变量 i （闭包），而定数处理函数被压入事件队列延迟执行，而执行的手 i 可能发生变化

改进的版本一

这个版本改进了上面的两个问题，解决方案也非常暴力，用 IFEE 在定义额外的作用域。
我们知道 javascript 语言默认是 function scope。解决问题二的思路就是在处理函数外定义一个 IFEE，把外部变量 复制 一份进来，从而形成一个新的作用域（变量）。这样处理函数回头执行的之后就不会找丢了。

改进的版本二

这个版本避免使用了丑陋的 IFEE，用 let 作为变量定义。

  1. let声明的全局变量不是全局对象的属性。这就意味着，你不可 以通过window.变量名的方式访问这些变量。它们只存在于一个不可见的块的作用域中，这个块理论上是Web页面中运行的所有JS代码的外层块。
  2. let声明的变量拥有块级作用域。也就是说用let声明的变量的作用域只是外层块，而不是整个外层函数。
  3. 形如for (let x…)的循环在每次迭代时都为x创建新的绑定。

改进版本三

是的，一行代码解决所有问题，这是函数式编程的魅力！

这个这个版本中，执行函数中引用的变量 e，是在当前迭代轮次的处理函数中的变量 e，每次迭代 e 的作用域都不是不一样的，因此 4个执行函数包含了 4作用域 e 变量的闭包，当然不会出问题了。

结论，多用 es6语法，但要知道它给我们带来的哪些好处。先苦后甜的道理在哪里都不会错。
