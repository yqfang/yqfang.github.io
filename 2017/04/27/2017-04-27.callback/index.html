很多人不明白回调，关于回调我希望用我自己理解的话做一个总结：

不阻塞当前线程，回过头来去执行某个方法。

传统同步遍历的困境

先来看一个例子: 假设现在有一个数组 a, 它有非常多的元素。

可以看到，只有当遍历完成后才能执行 doAnotherThing() 方法 假设 doAnotherThing(); 和 doSometingWithElem(elem); 两个方法并没有依赖关系，那么 我们并不希望 遍历并执行 doSometingWithElem(elem); 阻塞 doAnotherThing();

setTimeout 的思考

换句话说 在主线程结束之后再去执行一系列的 doWithElem 方法，这种避免主线程阻塞而异步执行的函数我们称之为『回调』

在主线程方法 main 执行完成之后 局部变量 elem 应该已经结束其生命周期了 但是由于被传入了 doWithElem 中，可以理解为这个变量的生命在另外一个执行上下文的得到了延续。这个方式可以理解为闭包。

所以我们经常有一种说法：javascript 是回调和闭包两个概念一般是成双成对的。

这里我们遇到了 javascript 十大经典错误之一，理解这个错误将有助于我们彻底理解闭包。

谈一下我的理解：

每次运行到 setTimeout 都会在把函数变量和函数中用到的外部变量生成一个执行上下文加入 事件队列

这里要注意的是传的变量都是变量的引用变量。这样就理解了上面代码的错误原因了。

setTimeout 把函数变量
function() {
callback(elem, counter, that);
} 和 elem, counter， that 变量的引用传入事件队列了

elem 和 counter 最终的值在主线程执行完成之后才确定。
事件队列中的事件处理函数和其他变量的值也由此确定。

看到这里你是不是明白了人们常说的：javascript 的回调是通过闭包实现的意思了
